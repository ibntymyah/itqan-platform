# إتقان — Itqan Platform

سقالة (scaffold) تأسيسية لـ **Phase 1 MVP** فقط، طبقاً للقسم 26 و35 من تعليمات المشروع. هذا ليس المنصة الكاملة بـ٢٢ وحدة — بل الأساس الذي تُبنى فوقه بقية المراحل بشكل تدريجي وقابل للمراجعة، تماشياً مع القسم 30 ("Do not attempt to build the entire system in one step").

## الدفعة الثانية: إعداد المؤسسة والبنية الأكاديمية

أُضيف الآن تحت `/settings`:

| المسار | الوظيفة |
|---|---|
| `/settings/branches` | إنشاء وعرض الفروع |
| `/settings/academic-years` | إنشاء سنوات دراسية، وفصول متداخلة تحت كل سنة |
| `/settings/study-periods` | العصر/المغرب/العشاء وترتيبها |
| `/settings/levels` | المستويات وترتيبها |
| `/settings/circles` | إنشاء حلقة (فرع+فصل+فترة+مستوى) وتعيين معلمين لها (يدعم أكثر من معلم) |

كل الكتابة تمر عبر `src/lib/actions/academic-structure.ts` (server actions)، وكل عملية:
1. تتحقق من الجلسة والصلاحية عبر `assertCan()`.
2. تتحقق أن **كل** مفتاح أجنبي مُدخَل (فرع/فصل/فترة/مستوى/موظف) يخص فعلياً مؤسسة المستخدم الحالي — قبل الكتابة، وليس فقط `org_id` المباشر على الكيان الجديد. هذا هو التطبيق العملي للقسم 22 ("لا يُعتمد على تصفية الواجهة الأمامية وحدها").
3. تُسجَّل في `audit_logs` (before/after JSON).

### الاختبارات

```bash
npm test                 # اختبارات وحدة — لا تحتاج قاعدة بيانات، تعمل الآن فعلياً (17/17 ناجحة)
npm run test:integration # اختبار عزل المؤسسات الفعلي — يحتاج DATABASE_URL حقيقياً
```

اختبار عزل المؤسسات (`tests/integration/tenant-isolation.test.ts`) يثبت أن مشرف مؤسسة (ب) لا يستطيع إنشاء حلقة أو فصل دراسي يُشير إلى موارد مؤسسة (أ) حتى لو كانت المعرّفات صحيحة الشكل، ولا تُظهر له أي قوائم بيانات مؤسسة أخرى. **لم يُشغَّل هذا الاختبار داخل بيئة المراجعة** لعين السبب الشبكي المذكور أعلاه — شغّله محلياً بعد إعداد قاعدة بيانات.

### ملاحظة عن أخطاء TypeScript المتبقية

`npx tsc --noEmit` يُظهر حالياً ١٤ خطأ من نوع "implicit any" في صفحات `/settings/*` — كلها ناتجة عن أن `@prisma/client` لم يُولَّد فعلياً هنا (لنفس سبب حجب الشبكة)، فتُرجع استعلامات Prisma النوع `any` بدل الأنواع المُولَّدة تلقائياً. هذه ليست أخطاء منطقية في الكود — **شغّل `npx prisma generate` محلياً وستختفي تلقائياً**. لن أضيف تعليقات نوع يدوية بديلة لأنها ستكرر ما يُولّده Prisma أصلاً وقد تنحرف عنه لاحقاً.

## ما لم يُبنَ بعد (عمداً)

كل ما هو خارج نطاق Phase 1 حسب القسم 4 من `itqan-phase1-architecture.md`: تسجيل الطلاب الكامل، الحضور والتسميع اليومي، تعيين/نقل الحلقات، تقارير الطلاب والحلقات، سجل التدقيق الفعلي، وكل الوحدات المؤجَّلة (المالية، النقاط، الرسائل، الشهادات، الأنشطة...). هذه تُبنى في الخطوة التالية دفعة تلو دفعة، وليس كلها دفعة واحدة.

## قيد لم يُطبَّق بعد في مستوى قاعدة البيانات

Prisma لا يدعم **partial unique index** مباشرة في ملف schema. القاعدة رقم 2 ("طالب واحد نشط لكل فترة دراسية") تحتاج migration SQL يدوية بعد أول `prisma migrate dev`، مثال:

```sql
CREATE UNIQUE INDEX student_circle_assignments_one_active_per_student
ON student_circle_assignments (student_id)
WHERE ended_at IS NULL;
```

هذا موثَّق كتعليق داخل `schema.prisma` عند حقل `student_circle_assignments`.

## ملاحظة حول التحقق من الـ schema

بيئة التنفيذ التي بُنيت فيها هذه الملفات لا تصل إلى مستودع محركات Prisma الثنائية (`binaries.prisma.sh`)، لذا لم أتمكن من تشغيل `prisma validate` أو `prisma generate` فعلياً هنا. راجعتُ العلاقات يدوياً وصححتُ خطأ حقيقي كنت قد ارتكبته (عمود واحد لا يمكن أن يكون مفتاحاً أجنبياً لجدولين معاً). **الخطوة الأولى بعد فك الضغط: شغّل الأمرين التاليين محلياً للتأكد:**

```bash
npm install
npx prisma validate
npx prisma generate
```

## قرارات وافتراضات

- `ASSUMPTION`: بيانات الأجزاء (juz) في `prisma/seed.ts` تقريبية على مستوى السورة لا الآية — تحتاج تحققاً من مصدر موثوق قبل الإنتاج (تفصيل في تعليق أعلى الملف).
- الأدوار السبعة الأساسية (`SYSTEM_ROLES` في `src/lib/permissions.ts`) لا تُنشأ تلقائياً هنا لأنها مرتبطة بكل مؤسسة (`org_id`) — ستُبنى ضمن "سكربت تهيئة المؤسسة" (Org Onboarding) في الدفعة التالية.
- `DECISION REQUIRED` سابقاً (المحفّظ، SMS، البصمة، معدّ الخطط) — تم ترحيلها رسمياً لمراحل لاحقة كما هو موثّق في وثيقة العمارة، القسم 9-ج المحدَّث.

## الخطوة التالية المقترحة

الدفعة القادمة: تسجيل الطالب (بحث عن تكرار، البيانات الأساسية والإدارية، ربطه بولي أمر وبحلقة عبر `student_circle_assignments`) — تمهيداً لتدفّق الحضور والتسميع اليومي الذي راجعناه سابقاً كواجهة.
